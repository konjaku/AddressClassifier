## 一、修复回归 Bug（HANDOFF §10.1）

### 问题

`华润置地广场写字楼-地下停车场(出入口)` → 地铁站出入口 ❌

### 根因

`split_fields` 将含"写字楼"的长名称误判为纯地址（`detect_mode` 返回 `address_only`），全部内容进 `address`、`name` 置空。

`addr/classify.py` 原逻辑：
```python
entity = main_entity(name) if name else address
```

`name` 为空时 `address` 直接作 entity，**未经过 `main_entity()` 剥离括号后缀**。`(出入口)` 保留，"出入口"在词典中命中「地铁站出入口」。

### 修复

```python
entity = main_entity(name) if name else main_entity(address)
```

### 验证

- 33 tests passed
- 回归评测 100% (12/12)
- 案例结果：地下停车场 ✅

## 三、修复分类级别——确保所有输出都是三级分类

### 问题

44.9% 的分类输出是一级或二级分类（如「住、宿」「金融服务」「公司」「租赁服务」），不满足用户"必须全都变成三级分类"的要求。

### 根因分析

**原因一：跨级重名被高位截断**

`categories.xlsx` 有 **48 个跨级重名**（如「金融服务」同时出现在二级和三级，「汽车销售及服务」同时出现在一级、二级和三级）。

`addr/categories.py` 中 `load_catalog` 原逻辑：
```python
if name in out:
    continue  # 保留首条 = 较高层级
```
`categories.xlsx` 按 一级→二级→三级 排序，首条是较高层级，导致通名词典中「金融服务」「公司」「餐馆」等全部指向二级甚至一级。

**原因二：一级/二级分类作为通名源**

`addr/build_dict.py` 原逻辑遍历**所有级别**分类的「适用对象」描述来提取通名。例如一级分类「住、宿」描述含"适用对象：公寓、酒店…"，导致「公寓」→「住、宿」（一级分类）。

### 修复

| 文件 | 改动 |
|------|------|
| `addr/categories.py` | 删除 `if name in out: continue`，后出现的（更深级别）覆盖先出现的 |
| `addr/build_dict.py` | 加 `if c.level != "三级分类": continue`，只从三级分类提取通名 |
| `tests/test_categories.py` | 「金融服务」预期级别从二级改为三级 |
| `dict/tongming.auto.yaml` | 重建（基于新 catalog 规则，仅三级分类生成） |

### 效果对比（5000 数据）

| 级别 | 修复前 | 修复后 |
|------|--------|--------|
| 三级分类 | 3194 (55.1%) | 4008 (69.1%) |
| 二级分类 | 641 (11.1%) | **0** |
| 一级分类 | 187 (3.2%) | **0** |
| 已分类行三级占比 | 79.4% | **100%** ✅ |

剩余无级别的行全部是 review（待复核）——系统判不出来，自然无分类级别，正确。

### 验证

- 33 tests passed
- 回归评测 100% (12/12)
- 5000 和 625 数据中所有已分类行(gold+silver)均为三级分类
